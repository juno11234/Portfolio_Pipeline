## 1. 개요 — 왜

- **why**: 9인 협업(기획4·개발5)에서 **스킬 프레임워크와 사운드 시스템**을 맡았다. 스킬 **데이터(SO)는 팀원이, 실행 프레임워크·개별 실행 스킬은 본인이** 만드는 분업이라, 서로 직접 참조하지 않고도 붙는 **연결 규약**이 필요했다. 게다가 방치형 로그라이크라 **밸런스 테스트로 데이터 테이블이 매우 자주** 바뀐다 → 매번 참조가 풀린다. 그래서 목표는 ① 데이터↔실행을 **정수 ID 하나**로 매칭, ② 반복 임포트에도 연결이 유지되는 **에디터 자동화**, ③ 런타임 강화가 원본 에셋을 건드리지 않는 **원본 보호**, ④ 클립을 인스펙터에 일일이 꽂지 않고 팀이 재생만 호출하는 **데이터화된 사운드**였다.

- **evidence**: 코드·문서로 확인되는 문제 — 밸런스 테스트로 데이터 테이블이 반복 수정되어 **임포트 때마다 스킬 프리팹 참조가 해제**됨(수동 재연결 필요), 겹치는 효과음을 위해 `AudioSource` 풀 **10개** 상주(동시 발음 수 = 풀 크기), !!2차 빌드 후 안드로이드에서만 매니저 클래스 null 참조!!. «측정 수치(임포트 1회당 풀린 참조 개수·수동 재연결 소요 시간 등)가 있으면 보강 — 없으면 이 프로젝트는 성능 수치보다 협업 결합도·빌드 디버깅·구조 판단이 핵심이라는 걸로 간다»

## 2. 고민과 선택

### 결정 A: 데이터↔실행 결합 — 직접 참조/상속 vs `ISkillID` 정수 계약

- 선택지:
  - 1. 데이터 SO와 실행 프리팹이 서로를 **직접 참조**하거나 상속으로 묶기
  - 2. 양쪽 모두 `ISkillID`(int) 계약만 구현하고 **ID로만** 매칭 ← 채택
- 기준별 비교:

  | 기준 | 1. 직접 참조/상속 | 2. `ISkillID` 정수 계약 |
  |---|---|---|
  | 결합도 | 데이터↔실행이 서로를 앎 | 서로 모른 채 ID로만 만남 |
  | 담당자 협업 | 데이터·실행 담당이 얽힘 | 데이터는 팀원, 실행은 본인 — 규약만 공유 |
  | 매니저의 타입 인지 | 구체 타입을 알아야 연결 | 타입 몰라도 `ISkillID.SkillID`로 연결 |

- **왜 2**: 스킬 데이터 SO는 팀원이, 실행 프레임워크는 본인이 만드는 분업이었다. 서로 직접 참조하면 한쪽을 고칠 때 다른 쪽이 딸려온다. 그래서 데이터/실행 양쪽이 각자 `SetSkillID()`로 ==에셋에 박힌 `Skill_ID`를 런타임 `SkillID`로 복사==하게 하고, 매니저는 `obj is ISkillID id`로 구체 타입을 몰라도 같은 ID끼리 이어붙였다.

### 결정 B: 런타임 스킬 데이터 — 원본 SO 직접 사용 vs 복제본(`Instantiate`)

- 선택지:
  - 1. 원본 `ScriptableObject`를 런타임에서 직접 참조·강화
  - 2. 시작 시 `Instantiate`로 **복제본**을 만들어 강화는 복제본에만 적용 ← 채택
- 기준별 비교:

  | 기준 | 1. 원본 직접 | 2. 복제본 |
  |---|---|---|
  | 원본 에셋 | 런타임 강화가 에셋에 영구 반영 | 세션 한정, 원본 무영향 |
  | 증상 | 플레이 종료 후에도 값 잔존 | 다음 플레이에 초기 상태 복원 |

- **왜 2**: !!레벨업 테스트 중, 런타임에 바뀐 데이터가 플레이 종료 후에도 그대로 남는 버그!!를 만났다. SO는 에셋이라 원본이 메모리에 상주해 변경이 그대로 남는 게 원인이었다. `MonoBehaviour`가 씬에 복사본으로 로드되는 것처럼 ==원본 대신 `Instantiate` 복제본을 사용==하니 종료 시 폐기되어 원본 오염이 사라졌다. 이후 강화(`SkillEnchant`)도 복제본 스탯만 바꾸므로 프로젝트 원본은 안전하다.

### 결정 C: 겹치는 효과음 재생 — 단일 소스 vs Queue 풀 vs List 탐색 풀

- 선택지:
  - 1. 단일 `AudioSource` — 새 효과음이 이전 것을 끊음
  - 2. `AudioSource` 풀 + `Queue`(선입선출)로 순서대로 꺼내기
  - 3. `AudioSource` 풀 + `List` 순회, **`isPlaying == false`인 소스**만 골라 재생 ← 채택
- 기준별 비교:

  | 기준 | 1. 단일 | 2. Queue 풀 | 3. List 탐색 풀 |
  |---|---|---|---|
  | 동시 재생 | 불가(끊김) | 가능 | 가능 |
  | 클립 길이 편차 대응 | — | 맨 앞이 재생 중이면 못 꺼냄 | 노는 소스만 골라 확보 |

- **왜 3**: 효과음이 겹쳐도 서로 끊기지 않아야 해서 단일 소스는 탈락. 풀에서 꺼낼 때 클립 길이가 제각각이라 `Queue` 선입선출로는 맨 앞이 아직 재생 중일 수 있다. 그래서 미리 만든 풀을 ==순회하며 `isPlaying == false`인 소스에 실어 재생==했다(동시 발음 수 = 풀 크기 `10`).

### 결정 D: 임포트 대응 — 수동 연결/드래그 vs 에디터 자동화

- 선택지:
  - 1. 임포트/에셋 추가 때마다 스킬을 인스펙터에 **손으로 드래그·재연결**
  - 2. `#if UNITY_EDITOR` 자동화 — 폴더 스캔으로 스킬 SO **자동 수집** + ID 매칭으로 프리팹 참조 **자동 복구** ← 채택
- 기준별 비교:

  | 기준 | 1. 수동 | 2. 에디터 자동화 |
  |---|---|---|
  | 반복 비용 | 임포트마다 전부 수동 | 스캔·ID 매칭으로 무인 처리 |
  | 실수 여지 | 누락·오연결 가능 | 규약대로 일괄 |

- **왜 2**: 밸런스 테스트로 데이터 테이블이 매우 자주 바뀌어 임포트마다 참조가 풀렸다. 매번 손으로 잇는 건 큰 비용이라, ==에디터에서만 도는 자동화로 폴더를 스캔해 스킬 SO를 모으고, 프리팹 ID와 SO ID를 비교해 자동 연결==했다. 단, 이 수집은 `#if UNITY_EDITOR`라 빌드엔 포함되지 않으므로 **런타임 전 에디터에서 수집이 끝나 있어야** 한다.

> «추가 후보(있으면 위 블록 복제해 채우기)» —
> · BGM 데이터 노출: `Dictionary`는 인스펙터 직렬화 불가 → `BGMData[]`로 받아 시작 시 런타임 `Dictionary`로 매핑, `bgmSource.clip != clip` 가드로 같은 곡 재시작 방지.
> · 레벨업 강화: 버프 쿨타임을 **하한 0.1초로 클램프** — 0 이하로 내려가 즉시 재발동되는 것 방지.

## 본인이 확인·보강할 곳 (핸드오프 전)

- **evidence 수치**: 측정값이 있으면 채우기. 없으면 정성 근거로 유지(성능 프로젝트가 아님).
- **결정 A~D의 "왜"**: 초안은 문서에 적힌 판단을 옮긴 것. 실제 그 갈림길에서 **무엇을 보고 골랐는지**를 본인 목소리로 한 번 더 다듬기 — 여기가 면접관이 보는 지점.
- **귀속 유지**: 스킬 데이터 SO(`ActiveSkillSO`/`BuffSO`)와 개별 스킬 프리팹은 **팀원(dopajac)** 작업물. 프레임워크·실행 스킬(`AquaBall`)·사운드 시스템만 "내가 만든 것"으로 둘 것.
