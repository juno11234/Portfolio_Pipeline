## 1. 개요 — 왜

- **why**: 소울라이크 보스 AI는 단순 추격·공격이 아니라 **페이즈 전환 · 3타 콤보 · 스킬 분기 · 회피 유도**처럼 조합 가능한 복잡한 행동이 필요하다. 상태 패턴으로 다루면 행동이 하나 늘 때마다 연결할 전이 조건이 기하급수적으로 늘어난다. 그래서 **행동이 늘어도 트리 구조가 유지되는 BT + 값 변화를 이벤트로만 전파하는 단방향(MVVM · `Observable<T>`) 구조**를 핵심 목표로 잡았다. 최종 지향점은 "공격 템포를 데이터로 조절해 플레이어에게 읽고 회피할 긴장감을 주는 AI".

- **evidence**: 이 프로젝트의 문제 제기는 프로파일러 수치가 아니라 **구조가 부족할 때 결정적으로 재현되는 버그**다. 대표 두 건 — (1) !!콤보 끊김!! : 적이 콤보 도중 플레이어가 사거리를 벗어나면, 매 프레임 트리를 재탐색하는 BT 특성상 진행 중이던 콤보가 **즉시** 끊기고 추격으로 빠진다(조건 충족 시 100% 재현). (2) !!걷기 모션 잔류!! : 멈춰야 할 상황에서도 애니 `Vertical` 파라미터가 1로 남아 걷기 모션이 계속 출력. 두 버그 모두 "왜 일반 `Sequence`/`Selector` 만으로는 부족하고 구조를 다듬어야 했나"의 근거다.

> ⚠️ **본인 확인 필요** — ②의 evidence 가이드는 "로딩 15초" 같은 실측 수치를 요구하지만, 이 프로젝트는 GC 최적화가 아니라 **행동 정합성(behavioral correctness)** 프로젝트라 "프레임당 GC 196회 → 4회" 같은 before/after 수치가 없다. 행동 버그의 근거는 프로파일러 delta가 아니라 **재현 조건**이므로 위처럼 적는 게 정직한 형태다. 보스방 프레임타임·이벤트 큐 적체 등을 실제로 측정한 값이 있으면 여기 숫자로 보강하되, **없으면 지어내지 말 것.**

## 2. 고민과 선택

### 결정 1: 적 AI 구조 — 상태 패턴 vs Behaviour Tree

- 선택지:
  - A. 상태 패턴(플레이어와 동일 방식)
  - B. Behaviour Tree(재귀 `Evaluate()`) ← **채택**

  | 기준 | A. 상태 패턴 | B. Behaviour Tree |
  |---|---|---|
  | 행동 추가 시 확장성 | 상태가 늘수록 전이 조건이 폭발 | 노드만 붙이면 되고 트리 구조는 유지 |
  | 확률적 행동 선택 | 상태마다 분기 로직을 심어야 함 | `Selector_Random` 으로 자연스럽게 표현 |
  | 리프(행동 단위) 재사용 | 상태 간 재사용 어려움 | `nodeDict` 공용 리프를 여러 몬스터가 조립 |
  | 구현 난이도 | 간단 | 복잡(초기 비용 큼) |

- **왜 그것을 골랐나**: 몬스터는 "상황 판단 + 확률적 행동 선택"이 핵심인데 상태 패턴은 여기서 전이 조건이 폭발한다. BT는 초기 구현이 복잡한 대신 ==행동이 늘어도 트리 구조가 유지==되고, `EnemyAIBase` 의 `nodeDict` 공용 리프를 `SkeletonAI`/`BossMonsterAI` 가 각각 조립해 **같은 리프로 다른 몬스터**를 만든다. 플레이어는 입력이 명확해 FSM, 몬스터는 판단이 필요해 BT라는 **의도적 비대칭**을 택했다.

### 결정 2: 애니메이션 전이 방식 — Trigger/Bool 파라미터 vs CrossFade

- 선택지:
  - A. Animator Trigger/Bool 파라미터 전이
  - B. 코드 기반 `CrossFade` 전이 ← **채택**(복귀·블렌드트리는 부분 혼용)

  | 기준 | A. Trigger/Bool | B. CrossFade |
  |---|---|---|
  | 전이 조건 복잡도 | 파라미터↔코드 동기화·하드코딩 필요 | 코드 한 줄, 해시로 직접 지정 |
  | Trigger 꼬임 / ResetTrigger | 관리 부담 큼 | 없음 |
  | 흐름 시각화·추적 | Animator에서 한눈에 | 전이가 코드 전체에 흩어져 추적 힘듬 |
  | 비프로그래머 조절 | 가능 | 어려움 |
  | 종료 전이 | 자동 | 직접 구현 |

- **왜 그것을 골랐나**: Trigger 꼬임·`ResetTrigger` 관리와 파라미터 하드코딩 동기화를 없애려 ==CrossFade 주도==를 택했다. CrossFade의 단점(추적·종료 전이)은 이렇게 보완 — (1) 전이를 `PlayTargetAniClip()` **함수로 한 번 감싸** 사용 위치 추적을 쉽게 하고 각 상태가 자기 전이를 관리, (2) 종료 전이는 **복귀 시에만** Animator FSM에 단방향으로 연결, (3) 연속 파라미터가 필요한 블렌드 트리는 `SetFloat`, 상태 전환은 CrossFade로 역할을 분담.

### 결정 3: 히트박스 판정 타이밍 — Animation Events vs StateMachineBehaviour

- 선택지:
  - A. Animation Events
  - B. StateMachineBehaviour(`AttackTiming`) ← **채택**

  | 기준 | A. Animation Events | B. StateMachineBehaviour |
  |---|---|---|
  | 판정 구간 표현 | 단발(한 프레임)만 | 구간(start~end) 표현 용이 |
  | 재사용성 | 클립마다 일일이 재설정 | SMB 하나를 여러 곳에서 재사용(인터페이스 추상화) |
  | 타입 안전 | 메서드 이름 하드코딩(이름 바뀌면 깨짐) | `IAttackAble` 타입세이프 직접 호출 |
  | 코드 없이 튜닝 | 가능(에디터) | 인스펙터 슬라이더로 가능 |

- **왜 그것을 골랐나**: 코드 없이 타이밍만 잡을 거면 Animation Events가 편하지만 이름 하드코딩이 깨지기 쉽고 구간 판정이 어렵다. ==StateMachineBehaviour== 는 `OnStateUpdate` 에서 `startNormalizedTime`~`endNormalizedTime` **구간**으로 콜라이더를 켜고, `GetComponentInParent<IAttackAble>()` 로 플레이어·스켈레톤·보스를 **다형적으로** 호출한다 — 히트박스 활성 타이밍을 코드가 아니라 **애니 클립 상태**가 소유하게 됐다. 판정 구간 · 타입 안전 · 재사용을 근거로 SMB를 택했다.

### 결정 4: 카메라 벽 충돌 처리 — RayCast vs SphereCast

- 선택지:
  - A. RayCast
  - B. SphereCast ← **채택**

  | 기준 | A. RayCast | B. SphereCast |
  |---|---|---|
  | 벽 관통 방지 | 모서리·얇은 틈을 뚫어 화면 가림 | 부피가 있어 모서리도 걸림 |
  | 계산 형태 | 선(무부피) | 반지름 있는 구 |

- **왜 그것을 골랐나**: !!벽 관통!! — 보스방에서 카메라가 벽을 뚫고 나가 화면이 가려졌다. RayCast는 선이라 모서리를 통과한다. ==SphereCast== 로 카메라 Pivot 뒤로 반지름 있는 구를 쏴 벽에 닿으면 그 거리만큼 Pivot에 더해 부드럽게 보간 이동시켰다. 더해서 회전(`Update`)과 추종·충돌(`LateUpdate`) 시점을 분리해 **위치→회전→충돌** 순으로 확정 = 떨림 방지.

### 결정 5: UI · 데이터 아키텍처 — MVC vs MVVM

- 선택지:
  - A. MVC
  - B. MVVM + `Observable<T>` ← **채택**

  | 기준 | A. MVC | B. MVVM |
  |---|---|---|
  | 결합도 | Controller가 View·Model 모두 참조 → 강한 결합 | 이벤트 기반 단방향 바인딩 → 느슨한 결합 |
  | 상태 전파·확장 | 참조를 직접 물어야 함 | 이벤트 발행 하나로 알림, 다양한 곳에서 구독만 |
  | Unity 의존 격리 | 규칙이 View에 섞이기 쉬움 | `FighterViewModel` 이 순수 C#(MonoBehaviour 미상속) |

- **왜 그것을 골랐나**: HP·스태미나·보스 페이즈처럼 **한 값의 변화를 여러 곳(UI 바 · State · 파티클)이 봐야** 하는 구조라 참조를 직접 물면 결합이 커진다. ==MVVM + `Observable<T>`== 는 값이 바뀔 때 이벤트만 발행하고 관심 있는 쪽이 구독한다 — 기능 추가 시 **구독만 붙이면** 된다. 실제로 보스 2페이즈는 `CurrentHealth` 가 절반 이하가 될 때 `OnSecondPhase` 이벤트 하나로 스킬 분기 개방 + 파티클을 동시에 연동했다(느슨한 결합의 실이득).

### 결정 6: 콤보 끊김 수정 방식 — 전용 Sequence_Combo(체크 내장) vs 기억하는 범용 Sequence

- 선택지:
  - A. 사거리 체크를 전용 `Sequence_Combo` 에 내장(중간 체크 노드 제거)
  - B. 범용 `Sequence` 에 실행 위치 기억 기능 추가(`Sequence_Memory`) ← **채택**

  | 기준 | A. 전용 Sequence_Combo | B. Sequence_Memory |
  |---|---|---|
  | 콤보 유지 | O(첫 콤보에서만 체크) | O(Running 자식 인덱스 기억 → 재개) |
  | 모듈화·재사용성 | 사거리 체크가 콤보에 묶여 다른 노드에서 재사용 불가 | 사거리 체크를 별도 리프로 다시 분리 가능 |
  | 트리 범용성 | 콤보 전용 노드가 하나 더 생김 | 조합 노드 한 종으로 모든 연속 행동 커버 |

- **왜 그것을 골랐나**: !!콤보 끊김!! 의 1차 수정은 사거리 체크를 콤보 시퀀스에 내장하는 것이었는데, 이러면 **같은 체크를 다른 노드에서 재사용할 수 없는** 새 문제가 생겼다. 그래서 ==`Sequence_Memory`== 로 방향을 틀었다 — `_lastRunningChildIndex` 로 Running이던 자식 위치를 기억했다가 그 자리에서 재개하니, 진입 조건을 다시 통과할 필요 없이 진행 중인 타부터 이어져 콤보가 완성된다. 결과적으로 사거리 체크를 **별도 리프로 다시 분리**해 모듈화·재사용성을 살렸다. (중단되면 안 되는 연속 행동은 `Sequence_Memory`, 순수 조건 검사만 필요한 곳은 기존 `Sequence` 로 역할 구분.)

  - { title: **이벤트 하나로 보스 2페이즈**, body: MVVM의 느슨한 결합 덕분에 체력 절반 이벤트(`OnSecondPhase`) 하나로 스킬 분기 개방 + 파티클을 깔끔히 연동했다. 수신 측이 멱등이라 반복 발행돼도 안전하다. }

- **learnings 아쉬웠던 점 · 깨달은 점**:
  - { title: **같은 원리의 노드 중단을 서로 다르게 우회함**, body: `Sequence`(콤보 끊김)와 `Selector`(걷기 모션 잔류)는 **Running이던 자식을 정리할 방법이 없다**는 동일한 근본 원인이었다. 그런데 당시엔 각각 `Sequence_Memory`(위치 기억)와 `Leaf_Cleaner`(강제 초기화)로 **따로** 우회했다. 하나의 근본 해법(상위 조건이 바뀌면 하위 노드를 정리)이 있다는 걸 두 번째에서야 알아챘다. }
  - { title: **시각화의 가치**, body: 코드로는 추적이 힘들던 행동 트리의 문제 지점이, 기술문서용으로 그린 다이어그램에서는 한눈에 보였다. 처음부터 도식화하며 디버깅했다면 시간을 크게 줄일 수 있었을 것이다. }
  - { title: **데이터 주도 BT의 가능성**, body: 노드를 enum으로 분류하다 보니, 트리 구조 자체를 데이터 테이블로 빼면 코드 수정 없이 AI 패턴을 조립할 수 있겠다고 느꼈다. }
  - { title: **`Observable<T>` 의 숨은 박싱을 놓침**, body: 값 변경 감지를 `if (Equals(_value, value)) return;` 로 했는데, 이 정적 `object.Equals(object, object)` 는 값 타입을 인자로 받을 때 **박싱**한다. HP·스태미나(`int`)처럼 자주 바뀌는 `Observable<T>` 는 대입할 때마다 박싱으로 !!불필요한 GC 할당!! 이 생긴다. ==`EqualityComparer<T>.Default.Equals(_value, value)`== 로 바꾸면 값 타입은 박싱 없이 비교된다 — 제네릭 반응형 프로퍼티에서 `object.Equals` 를 무심코 쓰면 안 된다는 걸 뒤늦게 알았다. }
