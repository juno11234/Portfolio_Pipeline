## 1. 개요 — 왜 (양식 ①의 what 과 짝을 이룬다)

- **why**: 게임을 하며 "해보고 싶던" 이동 액션 — 벽 달리기 · 슬라이딩 · 활공 · 워프 — 을 직접
  구현하는 것, 그리고 **벡터 연산(외적 · 내적 · 투영)을 실제 조작감으로 옮기는 것**이 목표였다.
  컷씬은 코드로 타이밍을 맞추는 대신 **Timeline 으로 프레임 단위 연출**을 다뤄보고 싶었다.
- **evidence**: 이 프로젝트는 성능 문제를 고치려던 게 아니라 **기능 · 연출 학습**이 목표라,
  착수를 촉발한 단일 실측 수치는 없다. 대신 개발 중 관측된 **3종 스파이크를 프로파일러로 진단**한
  실측이 §3 결과에 있다(에디터 랙 66ms/15FPS, 씬 로드 시 `GC.Collect` ~191ms 등).
  «기능별 정량 목표를 하나라도 세웠다면 여기에 추가 — 예: "60FPS 유지"·"컷씬 전이 프레임 튐 0"»

> 이 프로젝트에는 GC 스파이크류의 "고칠 문제"가 선행하지 않았다. 억지 수치를 만들지 않고, 실측은 결과 절로 넘겼다.

## 2. 고민과 선택 (핵심 기능과 분리된 독립 섹션)

### 결정 1: 적 시야 판정 방식 — 정확한 계산 vs 대소 비교 최적화  · (양식 ① 기능 4 · `Enemy.WatchPlayer`)
- 선택지:
  - A. `Vector3.Distance` + `Vector3.Angle` — 실제 거리 · 각도를 직관적 · 정확하게 얻음. 내부적으로 **제곱근(sqrt) · 역코사인(acos)** 연산.
  - B. `sqrMagnitude` + `Vector3.Dot` — 제곱거리 대소 비교 + 전방벡터와 방향벡터의 내적을 미리 캐싱한 `Cos(시야각*0.5)` 와 비교. 제곱근 · 역삼각함수 없이 곱셈 · 덧셈만. ← 채택
- 기준별 비교:

  | 기준 | A (Distance / Angle) | B (sqrMagnitude / Dot) |
  |---|---|---|
  | 얻는 정보 | 실제 거리 · 각도(정확) | 대소 관계만(비교엔 충분) |
  | 연산 비용 | 제곱근 + acos(역삼각) | 곱셈 · 덧셈, 코사인은 `Awake` 1회 캐싱 |
  | 코드 가독성 | 높음(의미가 그대로 드러남) | 낮음(제곱 · 내적의 의도를 알아야 함) |
  | 호출 빈도 | 매 프레임 `Update`(상시 감지) | 동일 |

- **왜 그것을 골랐나**: 시야 판정은 "범위 안/밖"이라는 **대소 비교만** 필요하지, 정확한 거리 · 각도 값
  자체는 쓰지 않는다. 그런데 이 판정은 `Update` 에서 **매 프레임 돌아 상시 감지**한다. 값이 필요 없는
  곳에서 매 프레임 sqrt · acos 를 태우는 건 낭비라, 가독성을 조금 포기하더라도 ==곱셈 · 덧셈만으로
  끝나는 B== 를 택했다. 시야각 코사인은 상수라 `Awake` 에서 한 번만 캐싱했다.

### 결정 2: 벽 달리기 진행 방향 — 투영 vs 외적  · (양식 ① 기능 1 · `WallRunPlayerState`)
- 선택지:
  - A. **투영**(`ProjectOnPlane`) — 전방을 벽 평면에 투영해 방향을 구함. !!벽이 기울면(법선 Y≠0) 그 경사가 진행 방향에 섞여 위/아래로 새는 버그!!
  - B. **외적**(`Vector3.Cross(wallNormal, Vector3.up)`) — 벽 법선과 월드 업벡터를 외적. 벽 기울기와 무관하게 항상 업벡터에 수직인(=수평인) 진행 축을 얻고, 부호는 그 축과 모델 전방의 내적으로 결정. ← 채택
- 기준별 비교:

  | 기준 | A (투영) | B (외적) |
  |---|---|---|
  | 평지 벽 | 정상 | 정상 |
  | 기울어진 벽 | !!경사 따라 방향 틀어짐(대각선 이탈)!! | 항상 수평 유지 |
  | 조작감 | 벽 각도에 따라 흔들림 | 기울어져도 일관된 수평 이동 |

- **왜 그것을 골랐나**: 처음엔 직관적으로 투영으로 짰는데, **기울어진 벽에서 캐릭터가 경사를 따라
  위/아래로 새는 문제**가 나왔다. 원인은 투영이 "벽 평면 위 방향"을 주다 보니 벽 기울기가 진행
  방향에 섞여 든 것. ==외적 `Cross(법선, 업벡터)` 은 두 벡터에 동시에 수직인 축을 주므로, 벽이
  얼마나 기울든 항상 수평인 진행 축이 나온다.== 벽 타는 조작감은 기울어져도 수평이 유지돼야
  안정적이라 외적으로 바꿨다.
  > 참고: **투영 자체가 나쁜 게 아니라 문제가 달랐다** — 슬라이딩(경사 하강)에는 투영이 맞다(§3 깨달음).

### 결정 3: 보스 AI 구현 방식 — 단일 클래스 vs 상태 패턴  · (양식 ① 기능 4 · `Middle_BossStateMachine`)
- 선택지:
  - A. **단일 클래스**(enum + switch 한 덩어리) — 잡몹처럼 한 클래스에 6상태를 몰아넣음. 초기 작성은 적지만, 스킬을 늘릴수록 거대한 switch 가 비대해지고 패턴 추가 때마다 기존 코드를 수정해야 함.
  - B. **상태 패턴**(`IBossState` · 상태마다 클래스) — 클래스만 추가하면 기존 코드 수정 없이 확장되고, 상태마다 책임이 분리됨. ← 채택
- 기준별 비교:

  | 기준 | A (단일 클래스 enum-switch) | B (상태 패턴 `IBossState`) |
  |---|---|---|
  | 상태 수(6+) | 한 switch 에 비대 | 상태별 클래스로 분산 |
  | 신규 공격 패턴 추가 | 기존 switch 수정 | 클래스만 추가(기존 불변) |
  | 상태별 책임 | 한 클래스에 혼재 | 상태마다 분리 |
  | 초기 작성량 | 적음 | 많음(인터페이스 · 클래스) |

- **왜 그것을 골랐나**: 보스는 6상태 + 스킬 랜덤 분기(돌진/충격파) + 그로기/처형으로 상태가 많고,
  **처음부터 공격 패턴 확장을 염두**에 뒀다. 단일 클래스 enum-switch 면 스킬을 추가할 때마다
  거대한 switch 를 계속 수정해야 한다. ==상태 패턴은 클래스만 추가하면 기존 코드 수정 없이
  확장되고, 상태마다 책임이 분리돼 유지보수가 쉽다.== 복잡도와 확장성이 큰 보스라, 상태 패턴의
  이득이 초기 작성량을 넘어선다고 봤다.

### 결정 4: 잡몹 AI 구현 방식 — if-else vs enum-switch  · (양식 ① 기능 4 · `Enemy`)
- 선택지 (잡몹은 상태 4개라 상태 패턴까진 과함 — 한 클래스 안에서 전이 방식만 결정):
  - A. **if-else** — 조건문으로 상태 분기. 빠르게 짜지지만 상태 · 전이가 늘면 조건이 흩어져 지금 어떤 상태인지 한눈에 안 들어옴.
  - B. **enum + switch** — 상태를 `EnemyState` enum 으로 명시하고 switch 로 분기. 어떤 상태가 있고 각 상태에서 뭘 하는지 한눈에. ← 채택
- 기준별 비교:

  | 기준 | A (if-else) | B (enum-switch) |
  |---|---|---|
  | 초기 구현 속도 | 빠름 | 보통 |
  | 상태 가시성 | 조건에 흩어짐 | enum 으로 명시 |
  | 가독성(4상태) | 무난 | 더 명확 |
  | 상태 추가 | 조건 중첩 | `case` 추가 |

- **왜 그것을 골랐나**: 잡몹은 순찰 → 발견 시 공격 → 이탈 시 추격의 단순 4상태 루프라, 클래스를
  나누는 상태 패턴까진 과하고 **한 클래스 안에서 전이 방식만** 정하면 됐다. 처음엔 **if-else 로
  빠르게 짜 동작을 검증**하고, 규칙이 굳은 뒤 ==상태 구분이 명확한 enum-switch 로 리팩터링== 해
  가독성을 확보했다. "동작 검증 → 구조 정리" 순서로 재작업을 줄였다.

### 결정 5: QTE 키 입력 비교 — 문자열 비교 vs int 해시 비교  · (양식 ① 기능 5 · `QTEManager.OnActionPerformed`)
- 선택지:
  - A. 액션 이름(**string**)을 정답 문자열과 직접 비교 — 직관적이나 문자를 하나씩 대조.
  - B. `Animator.StringToHash` 로 **int 해시화**해 정답 해시 배열(`qteKeyHash`)과 비교. ← 채택
- 기준별 비교:

  | 기준 | A (string) | B (int 해시) |
  |---|---|---|
  | 비교 비용 | 문자 단위 대조 | int 1회 비교 |
  | 가독성 | 이름 그대로 | 값만 봐선 모름 |
  | 준비 | 없음 | 시작 시 정답을 해시로 캐싱 |

- **왜 그것을 골랐나**: 입력마다 정답과 비교하는데, 문자열은 매번 문자 단위로 대조해 상대적으로
  비싸다. **시작 시 정답 시퀀스를 한 번 해시로 만들어 두면 입력 비교가 int 한 번**으로 끝난다.
  QTE 는 짧은 시간에 입력이 몰려 비교가 잦으므로 ==해시 비교== 로 바꿨다.
  > 규모가 작아(키 몇 개) 체감 차이는 미미하다 — 면접에서 "측정했나"를 물으면 **정석 습관** 관점으로 답할 것.

## 3. 결과 · 회고

- **results** (성능 개선 수치가 아니라, 3종 스파이크가 **실제 병목인지 판별**한 것이 결과):
  - 에디터 재생 중단 랙(66ms/15FPS) — 원인 = `EditorLoop` 99.6%(에디터 전용 오버헤드) → 빌드엔 없음.
  - 씬 로드 시 튐 — 원인 = 씬 전환에서 Unity 내부가 `GC.Collect` 를 호출하며 걸리는 블로킹(`GC.Collect` ~191ms).
  - 지속적 작은 스파이크 — `Gfx.WaitForPresent`(~14.55ms) = GPU 병목이 아니라 **VSync 모니터 갱신 대기**, 빌드에서 정상 확인.
