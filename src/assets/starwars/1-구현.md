## A2. 시스템 구조 — 말로 서술

- **intro**: 입력 → 상태 머신 → 애니메이션 → 히트박스 → **이벤트 큐** → 데미지 처리로 이어지는 단방향 흐름이 골격이다. 전투 참여자는 모두 `IFighter` 로 추상화되고, 실제 피해 전달은 `CombatSysytem` 싱글턴의 큐가 프레임 단위로 중계한다.

### 계층 구성 (위 → 아래)

1. **입력 계층** — `InputManager`(Input System 액션맵 구독) → `PlayerStateMachine` 의 입력 프로퍼티(`MoveInput` · `JumpInput` · `AttackInput` · `GuardInput` …)에 값만 주입.
2. **상태 계층** — `PlayerStateMachine`(11개 `IPlayerState`) · `Middle_BossStateMachine`(6개 `IBossState`) · `Enemy`(enum 4상태). 각자 상태 머신을 굴린다.
3. **감지 계층** — `Blade` · `Bullet` · `Middle_DashAttackColl`(콜라이더 트리거) 및 `PlayerAttackSender` · `BossAttackSender`(`StateMachineBehaviour`). 충돌/애니 타이밍을 잡아 `CombatEvent` 를 생성.
4. **전투 계층** — `CombatSysytem`(싱글턴) 이 `CombatEvent` 큐를 프레임당 최대 10개 처리하고 `IFighter.TakeDamage` 를 호출.
5. **연출 계층** — `CutsceneManager` · `QTEManager` · `QTETimelineBranch` · `BossCameraShakeTrigger` / `FlyCameraShakeTrigger`(Cinemachine) · `Portal`.
6. **지원 계층** — `SoundManager`(AudioSource 오브젝트 풀 싱글턴 · `DontDestroyOnLoad`).

- **구성요소(무엇으로)**: Input System 액션맵, 두 방식의 상태 머신(**클래스 State 패턴** = Player·Boss / **enum switch** = 일반 적), `CombatSysytem` 이벤트 큐, `IFighter` 계약, Timeline · Cinemachine, `SoundManager` 오브젝트 풀.
- **관계(누가 누구를 아는가)**: 감지 콜라이더(`Blade`/`Bullet`/`Middle_DashAttackColl`)와 `AttackSender` 가 `CombatSysytem` 에 이벤트를 **등록만** 하고, `CombatSysytem` 이 `IFighter.TakeDamage` 를 **호출**한다(송신자는 수신자를 직접 모른다). 상태 클래스들은 오직 자신의 StateMachine 을 통해 전이한다. 전역 참조는 `Player.CurrentPlayer`(static) 와 각 매니저 싱글턴뿐이다. 넉백은 `KnockbackClass.KnockbackFunc`(static `Action<Vector3>`)로 호출해 결합을 끊었다.
- **흐름(입력 → … → 출력)**: 입력 → 상태 전환 → 애니메이션 재생 → `AttackSender` 가 `normalizedTime` 구간에 히트박스 On → 콜라이더 충돌 → `CombatEvent` 큐 등록 → `CombatSysytem.Update` 가 최대 10개 dequeue → `TakeDamage`(hp 차감 · 사망 · 그로기) → (보스) 그로기/사망 시 QTE·타임라인 연출.

### 핵심 클래스 관계

- **`IFighter`** — `MainCollider` · `GameObject` · `Attack()` · `TakeDamage(CombatEvent)`. **`Player` · `Enemy` · `Middle_BossStateMachine` 이 모두 구현**하는 "전투 참여자" 계약. `CombatSysytem` 은 `Dictionary<Collider, IFighter>` 로 콜라이더에서 역참조한다.
- **`IPlayerState` / `IBossState`** — State 패턴 계약. `IPlayerState` = `Enter/Input/UpdateLogic/Exit`, `IBossState` = `Enter/UpdateLogic/Exit`. 각 상태가 클래스로 구현되고, StateMachine 이 `currentState` 하나만 들고 `ChangeState` 로 `Exit → 교체 → Enter`.
- **`InGameEvent`(추상) → `CombatEvent`** — 이벤트 계층. `EvenetType` 으로 분기하며 현재 `Combat` 타입만 사용(확장 여지 있는 큐 구조).
- **`StateMachineBehaviour` 상속** — `PlayerAttackSender` · `BossAttackSender`. Animator 상태에 직접 부착되어 애니 진행도로 히트박스를 여닫는다.

## A3. 핵심 기능

### 기능 1: 벡터 수학 기반 파쿠르 이동 (플레이어 상태 머신)

- **무엇을 · 어떻게 만들었나**
  - **상태 분리**: 이동/전투를 `PlayerStateType` 11종(Move · Jump · WallRun · WallJump · Sliding · Flying · Attack · Guard · Knockback)으로 나눠 각각 `IPlayerState` 클래스로 구현. `PlayerStateMachine.Update` 는 중력·`controller.Move` 만 공통 처리하고 나머지는 현재 상태에 위임한다.
  - **벽 달리기(외적 · 내적)**: 벽 법선과 업벡터를 **외적**해 벽을 따라갈 방향을 얻고, 그 방향과 모델 전방을 **내적**해 부호(앞/뒤)를 정한다. 외적을 쓰면 벽이 기울어도 항상 업벡터에 수직인 진행 방향이 나온다.
  - **슬라이딩(투영)**: 바닥 법선을 레이캐스트로 구하고 다운벡터를 그 평면에 **투영**(`ProjectOnPlane`)해 경사면을 타고 내려가는 방향을 만든다.
  - **활공(Flying)**: `Portal` 진입 시 `waypoints` 를 받아 웨이포인트를 순차 추종, 마지막 지점에서 Jump 상태로 이탈.
  - **이중 점프 · 벽 점프**: `maxJumpCount` 로 점프 횟수 제한, 벽 점프는 법선 방향으로 수평 임펄스.
- **핵심 코드**

  벽 달리기 — 외적으로 진행 방향, 내적으로 부호 결정 (`WallRunPlayerState.UpdateLogic`, 코드 주석 `// 외적` · `// 방향 조절` 그대로):
  ```csharp
  // 외적
  Vector3 moveDir = Vector3.Cross(wallNormal, Vector3.up).normalized;
  // 방향 조절
  if (Vector3.Dot(moveDir, player.Model.forward) < 0) moveDir = -moveDir;
  // 모델 회전
  player.Model.rotation = Quaternion.LookRotation(moveDir, wallNormal);
  // 이동
  player.Controller.Move(moveDir * (wallRunSpeed * Time.deltaTime));
  ```
  > **어디가 · 왜**: `Cross(wallNormal, up)` 은 벽이 기울어 법선의 Y가 0이 아니어도 항상 업벡터에 수직인 벡터를 주므로 대각선으로 새지 않는다. `Dot(..) < 0` 은 벽을 따라가는 두 방향 중 플레이어가 보던 쪽을 고른다.

  슬라이딩 — 바닥 법선에 다운벡터 투영 (`SlidingPlayerState.UpdateLogic`):
  ```csharp
  Vector3 downDir = Vector3.ProjectOnPlane(Vector3.down, slopeNormal).normalized;
  Vector3 move = downDir.normalized * (player.slidingSpeed * Time.deltaTime);
  player.Controller.Move(move);
  ```
  > **어디가 · 왜**: 다운벡터를 경사면(법선)에 투영하면 경사 방향 성분만 남아 벽/바닥을 파고들지 않고 표면을 타고 내려간다.
- **필요한 그래프**: **상태 머신(FSM)** — "어떤 입력·조건이 어떤 이동 상태로 전이시키나?"
  - 노드(11): Move · Jump · WallRun · WallJump · Sliding · Flying · Attack · Guard · Knockback (+ 진입점 Move)
  - 엣지(전이 조건):
    - Move → Attack(`AttackInput`) · Guard(`GuardInput` & 스태미나>0) · Jump(`JumpInput`) · Sliding(`SlidingTrigger`) · Flying(`FlyingTrigger`)
    - Jump → WallRun(벽 감지 & `RunInput`) · Move(착지 & 진입 딜레이 경과)
    - WallRun → WallJump(`JumpInput` 또는 벽 이탈)
    - WallJump → Jump(`TryJump`) · WallRun(벽 감지 & `RunInput` & 쿨) · Move(착지)
    - Sliding → Jump(`SlidingTrigger` 해제)  |  Flying → Jump(웨이포인트 끝)
    - Attack → Move(공격 애니 `normalizedTime` ≥ 0.95); Attack1→2→3 콤보(`AttackInput`)
    - Guard → Move(가드 해제 또는 스태미나 소진)
    - Knockback → Move(2초 경과); 진입은 `KnockbackClass.KnockbackFunc`
- **표 · 카드로 낼 사실** (플레이어 이동 스탯, `PlayerStateMachine`):

  | 항목 | 값 | 항목 | 값 |
  | --- | --- | --- | --- |
  | 걷기 속도 `walkSpeed` | 5 | 최대 점프 횟수 `maxJumpCount` | 2 |
  | 달리기 속도 `runSpeed` | 9 | 슬라이딩 속도 `slidingSpeed` | 50 |
  | 점프 높이 `jumpHeight` | 3 | 벽 점프 수평 속도 `wallJumpHorizontalSpeed` | 100 |
  | 중력 `gravity` | -9.81 | 활공 속도 `flySpeed` | 20 |

### 기능 2: 이벤트 큐 기반 전투 시스템 (`CombatSysytem` · `IFighter`)

- **무엇을 · 어떻게 만들었나**
  - **큐 중계**: 모든 피해는 `CombatEvent` 로 만들어 `CombatSysytem` 싱글턴 큐에 넣고, `Update` 에서 **프레임당 최대 `Max_Event_COUNT`(=10)개**만 꺼내 처리한다. 대량 동시 타격의 처리 스파이크를 상한으로 눌렀다.
  - **송수신 추상화**: 송신자/수신자를 `IFighter` 로 잡아 서로의 구현을 모른 채 피해를 주고받는다. `Dictionary<Collider, IFighter>` 로 충돌한 콜라이더에서 대상을 역참조한다.
  - **애니 동기 히트박스**: `PlayerAttackSender`(`StateMachineBehaviour`)가 공격 애니의 `startNormalizedTime`~`endNormalizedTime` 구간에만 검(`Blade`) 콜라이더를 켠다. 판정이 애니 모션과 정확히 맞물린다.
- **핵심 코드**

  큐 처리 상한 (`CombatSysytem.Update`):
  ```csharp
  int processCount = 0;
  while (inGameEventQueue.Count > 0 && processCount < Max_Event_COUNT)
  {
      var inGameEvent = inGameEventQueue.Dequeue();
      switch (inGameEvent.Type)
      {
          case InGameEvent.EvenetType.Combat:
              var combatEvent = inGameEvent as CombatEvent;
              inGameEvent.Reciever.TakeDamage(combatEvent);
              break;
      }
      processCount++;
  }
  ```
  > **어디가 · 왜**: `processCount < Max_Event_COUNT` 로 한 프레임 처리량을 10개로 제한 — 남은 이벤트는 다음 프레임으로 이월돼 순간 부하를 평탄화한다.

  애니 타이밍에 히트박스 On/Off (`PlayerAttackSender.OnStateUpdate`):
  ```csharp
  if (passStartNormalizedTime == false && startNormalizedTime < stateInfo.normalizedTime)
  {
      Player.CurrentPlayer.AttackCollOn();
      passStartNormalizedTime = true;
  }
  if (passEndNormalizedTime == false && endNormalizedTime < stateInfo.normalizedTime)
  {
      Player.CurrentPlayer.AttackCollOff();
      passEndNormalizedTime = true;
  }
  ```
  > **어디가 · 왜**: `pass*` 플래그로 각 시점을 프레임당 1회만 발동시켜, 애니 진행도 기준의 히트박스 개폐를 중복 없이 처리한다.
- **필요한 그래프**: **흐름도(flowchart)** — "한 번의 타격이 데미지가 되기까지 무엇을 거치나?"
  - 노드/엣지: 공격 입력 → `AttackPlayerState`(애니 Trigger) → `AttackSender` 가 `normalizedTime` 구간에 히트박스 On → `Blade.OnTriggerEnter`(Enemy 태그) → `CombatEvent` 생성 · `AddInGameEvent` → `CombatSysytem.Update` 가 최대 10개 dequeue → `IFighter.TakeDamage` → hp 차감/사망
- **표로 낼 사실**:

  | 값 | 수치 | 출처 |
  | --- | --- | --- |
  | 프레임당 이벤트 처리 상한 `Max_Event_COUNT` | 10 | `CombatSysytem` |
  | 플레이어 검 데미지 `Blade.damage` | 35 | `Blade` |
  | 일반 적 총알 데미지 `Bullet.damage` | 10 | `Bullet` |
  | 보스 돌진 데미지 `Middle_DashAttackColl.damage` | 20 | `Middle_DashAttackColl` |

### 기능 3: 가드 · 패링 시스템 (스태미나)

- **무엇을 · 어떻게 만들었나**
  - **패링 윈도우**: Guard 상태 진입 직후 **0.3초** 동안 `IsParrying`. 이 사이 근접 피해는 무효화되고 패링 파티클을 재생한다.
  - **총알 반사**: 패링 중 총알(`Bullet`)에 맞으면 데미지 대신 진행 방향을 반전(`parryBullet = true`)시켜 되돌리고, 반사된 총알이 적에 맞으면 이번엔 적이 `CombatEvent` 를 받는다.
  - **가드 흡수**: 패링 윈도우가 지나도 가드 중이면 피해를 hp 대신 **가드 스태미나**에서 깎고, 0이 되면 `GuardBreak`. 스태미나는 가드 쿨(`guardCool` 2초) 이후 1초당 1씩 자연 회복.
- **핵심 코드**

  피해 수신 분기 (`Player.TakeDamage`):
  ```csharp
  if (IsParrying)
  {
      parryingParticle.Play(true);
      return;
  }
  if (IsGuarding)
  {
      stats.guardStamina -= combatEvent.Damage;
      if (stats.guardStamina <= 0) GuardBreak();
      return;
  }
  stats.hp -= combatEvent.Damage;
  if (stats.hp <= 0) Die();
  ```
  > **어디가 · 왜**: 패링 → 가드 → 일반 피격 순으로 우선순위를 두어, 같은 `CombatEvent` 라도 플레이어 상태에 따라 무효/스태미나/체력으로 갈라 처리한다.
- **필요한 그래프**: 표/알약(카드)로 충분(분기 3갈래). 그래프 불필요.
- **표로 낼 사실**:

  | 피격 시 상태 | 처리 | 관련 수치 |
  | --- | --- | --- |
  | 패링(진입 0.3초 내) | 근접 무효 · 총알 반사 | `parryTime` 0.3s |
  | 가드 중 | 스태미나 차감, 0 → GuardBreak | `guardStamina` 50 |
  | 그 외 | hp 차감, 0 → 사망 | `hp` 100 |
  | 스태미나 회복 | 가드 쿨 후 1초당 +1 | `guardCool` 2s |

### 기능 4: 이원화된 적 · 보스 AI

- **무엇을 · 어떻게 만들었나**
  - **일반 적(`Enemy`)** — enum + switch FSM(Patrol · Chase · Attack · Die). NavMesh 로 순찰/추격하고, 시야는 각도(내적)와 거리로 판정한 뒤 레이캐스트로 차폐를 확인한다. 시야각 코사인은 `Awake` 에서 한 번 캐싱한다.
  - **중간보스(`Middle_BossStateMachine`)** — `IBossState` 클래스 State 패턴 6종(Chasing · NormalAttack · PulseAttack · DashAttack · Groggy · ExcutionReady). 스킬 쿨(15초)이 차면 돌진/장판을 **랜덤 선택**한다. **돌진이 벽(`Obstacle`)에 부딪히면 그로기(3초)** 에 빠져 처형 기회를 준다.
  - **최종보스(`FinalBossAnimatorController`)** — 자체 상태 머신 없이 Animator 트리거 래퍼(Hide/Appear/Scream/Attack/…)만 노출. 타임라인·시그널로 구동되는 연출형 보스.
- **핵심 코드**

  시야 판정 — 내적으로 각도, 캐싱한 코사인 비교 (`Enemy.WatchPlayer`):
  ```csharp
  float dotProduct = Vector3.Dot(transform.forward, direction);
  float halfViewAngleCosine = _halfViewAngleCosine; // Awake 에서 Cos(viewAngle*0.5) 캐싱
  if (dotProduct < halfViewAngleCosine || distance > stats.viewDistance) return false;
  Physics.Raycast(transform.position + Vector3.up, direction, out RaycastHit hit, stats.viewDistance);
  ```
  > **어디가 · 왜**: 각도 비교를 `Vector3.Angle` 대신 **내적 vs 미리 구한 코사인**으로 처리 — 매 프레임 역삼각함수를 피한다. 통과한 경우에만 레이캐스트로 벽 뒤 은폐를 확인한다.

  스킬 랜덤 선택 (`Middle_BossStateMachine.UseSkill`):
  ```csharp
  int random = Random.Range(0, 2);
  if (random == 0) { skillState = new Middle_DashAttackState(this); skillStateType = MiddleBossStateType.DashAttack; }
  else            { skillState = new Middle_PulseAttackState(this); skillStateType = MiddleBossStateType.PulseAttack; }
  ```
  > **어디가 · 왜**: 쿨(`skillCool` 15s)이 찰 때마다 두 스킬 중 하나를 뽑아 패턴을 예측 불가능하게 만든다.
- **필요한 그래프**: **상태 머신(FSM) 2개**
  - 일반 적 — "적이 언제 순찰/추격/공격/사망으로 바뀌나?" 노드(4): Patrol · Chase · Attack · Die / 엣지: Patrol→Chase(플레이어 시야 확보 또는 피격) · Chase→Attack(사거리 & 시야) · Chase→Patrol(이탈) · Attack→Chase(발사 애니 종료/이탈) · any→Die(hp≤0)
  - 중간보스 — "보스가 언제 스킬/공격/그로기/처형으로 가나?" 노드(6): Chasing · NormalAttack · PulseAttack · DashAttack · Groggy · ExcutionReady(Dead) / 엣지: Chasing→NormalAttack(거리<6.5) · Chasing→Dash|Pulse(쿨 15s, 랜덤) · Dash→Groggy(벽 충돌 `CrashWall`) · Pulse/NormalAttack→Chasing(애니 종료) · Groggy→Chasing(3s) · any→ExcutionReady(`OnDie`)
- **표로 낼 사실**:

  | 대상 | 값 | 수치 |
  | --- | --- | --- |
  | 일반 적 | hp / 시야각 / 시야거리 / 추적범위 | 100 / 120° / 10 / 30 |
  | 중간보스 | hp / 스킬 쿨 / 근접 전환 거리 / 그로기 | 300 / 15s / 6.5 / 3s |

### 기능 5: 옵저버 패턴 QTE + 타임라인 · 카메라 연출

- **무엇을 · 어떻게 만들었나**
  - **QTE(옵저버)**: `QTEManager` 가 별도 `QTE` 액션맵을 켜고 입력을 구독, 눌린 액션명을 `Animator.StringToHash` 로 정답 시퀀스와 비교한다. 제한 시간 안에 순서대로 다 누르면 성공, 틀리거나 시간 초과면 실패 — 결과를 `Action`(`OnQTESuccess` · `OnQTEFailed`) 이벤트로 발행한다. 구독자는 QTE 내부 구현을 모른다.
  - **타임라인 분기**: `QTETimelineBranch` 가 `OnQTEFailed` 를 구독해 진행 중 타임라인을 멈추고 실패 타임라인을 재생한다. QTE 는 중간보스 그로기/처형 연출과 맞물린다.
  - **컷씬 · 카메라**: `CutsceneManager` 가 컷씬 프리팹을 `Instantiate` 하고 `PlayableDirector.Play`. `BossCameraShakeTrigger`/`FlyCameraShakeTrigger` 가 Cinemachine `ImpulseSource` 로 피격·보스 등장·벽 충돌 시 화면을 흔든다.
  - **활공 트리거**: `Portal` 진입 시 카메라를 흔들고 `waypoints` 를 넘겨 `FlyingTrigger` 를 세워, 플레이어를 Flying 상태(웨이포인트 추종 비행)로 진입시킨다.
- **핵심 코드**

  입력 vs 정답 해시 비교 (`QTEManager.OnActionPerformed`):
  ```csharp
  var actionNameHash = Animator.StringToHash(context.action.name);
  if (actionNameHash == qteKeyHash[currentIndex])
  {
      MarkKeySuccess(currentIndex);
      currentIndex++;
      if (currentIndex >= qteKey.Count) EndQTE(true);
  }
  else { MarkKeyFail(currentIndex); EndQTE(false); }
  ```
  > **어디가 · 왜**: 문자열 대신 해시로 비교해 매 입력 비교 비용을 낮췄다. `currentIndex` 가 시퀀스 끝에 닿으면 성공 종료, 하나라도 틀리면 즉시 실패 종료.

  실패 이벤트 구독 → 타임라인 분기 (`QTETimelineBranch`):
  ```csharp
  private void Start() { qteManager.OnQTEFailed += QTEFail; }
  public void QTEFail() { currentTimeline.Stop(); failTimeline.Play(); }
  ```
  > **어디가 · 왜**: QTE 는 결과만 이벤트로 던지고(발행), 연출 전환은 구독자가 책임진다 — QTE 로직과 연출이 분리된 느슨한 결합.
- **필요한 그래프**: **흐름도(flowchart)** — "QTE 입력이 어떤 연출로 갈라지나?"
  - 노드/엣지: `QTETrigger`(트리거 진입) → `QTEManager.StartQTE`(QTE 액션맵 Enable · UI 생성) → 입력마다 해시 비교 → [순서 완주] → `OnQTESuccess`  |  [오입력/시간초과] → `OnQTEFailed` → `QTETimelineBranch` 가 현재 타임라인 Stop · 실패 타임라인 Play
