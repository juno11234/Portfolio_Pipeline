## 1. 개요 — 왜 (양식 ①의 what 과 짝을 이룬다)
- **why**: «대량의 오브젝트를 안정적으로 처리하기 위한 GC 최적화에 초점을 맞췄습니다.»
- **evidence**: «• GC Alloc In Frame : 196회 → 4회, (97% 감소), • Materials : 824회 → 249회, (70% 감소),메모리 사용량 140MB 절감»

> 숫자 없는 문제 제기는 주관적 감상이라 근거가 못 된다. 실측값을 적는다.

## 2. 고민과 선택 — 최소 1건 (핵심 기능과 **분리된 독립 섹션**. 없으면 빌드가 깨진다)
> 기능이 많아 기능 섹션을 여러 개로 쪼개도, 결정은 기능 안에 섞지 않고 **여기 한데 모은다**(뱀서식 '04 고민과 선택').
> "'그냥'은 금지어. 신입에게 기대하는 건 기술력이 아니라 사고 과정."
> 결정마다 아래를 반복한다.

### 결정: «무엇을 두고 고민했나. 예: 전투 처리 — 직접 호출 vs 이벤트 큐»
- 선택지 (2개 이상, **정확히 하나만** 채택):
  - A. «…»
  - B. «…» ← 채택
- 기준별 비교 (**모든 선택지를 같은 잣대로** 채운다 — 빈칸은 비교가 아니다):

  | 기준        | 직접 호출 | 중앙처리 이벤트큐 |
  |-----------|----|-------|
  | «결합도 · 확장성» | «공격자가 대상을 직접 호출, 스킬 추가할 때마다 공격 처리지점이 분산» | «AddInGameEvent만 알면 됨, 신규 스킬 추가해도 무변경, 한 곳에서 확장»    |
  | «처리 타이밍»  | «같은 프레임 즉시 반영» | «최대 1프레임 지연»    |
  | «반환값 활용»  | «TakeDamage의 실제 데미지·오버킬·사망여부를 즉시 리턴받아 처리 가능» | «리턴값 없음 → 역이벤트/콜백 필요»    |
- | «순회·재진입 안전성»  | «타격 중 사망→컬렉션 변경을 유발» | «때리는 시점 != 죽는 시점 으로 분리»    |
- **왜 그것을 골랐나**: «직접 호출은 '타격 도중 적 사망 → 풀 반납·컬렉션 변경' 이 중첩돼 오류 위험, 이벤트 큐는 이를 구조적으로 차단하고느슨한 결합과 확장을 얻음. 처리 타이밍 문제는 미미하고 반환값 활용은 콜백으로 보완 가능하다 판단하여 B 결정»

- 선택지 (2개 이상, **정확히 하나만** 채택):
    - A. «…»
    - B. «…» ← 채택
- 기준별 비교 (**모든 선택지를 같은 잣대로** 채운다 — 빈칸은 비교가 아니다):

  | 기준            | OnTriggerEnter2D | Raycast & OverlapCircle |
    |---------------|--|--|
  | «터널링 대응»      | «빠른 투사체가 콜라이더를 통과하면 놓침» | «Raycast로 이동 궤적 검사 → 놓치지 않음» |
  | «대량 처리·물리 부하» | «한쪽에 Rigidbody2D 필수 → 수백 개면 부하» | «바디 불필요 + 사전 할당 버퍼로 매 프레임 GC 0» |
- | «충돌 정밀도»      | «콜라이더 실제 형상 기반 정밀 판정» | «OverlapCircle은 원으로 근사» |
- | «구현비용»            | «콜백만 구현 충돌 드물면 저렴» | «버퍼·필터·중복판정 직접 관리, 매 프레임 쿼리» |
- **왜 그것을 골랐나**: «정밀한 히트박스가 중요하지 않고 대량 충돌이 많이 일어나는 뱀서라이크 장르 특성상
  정밀한 콜라이더 형상으로 Rigidbody 부하가 있는 OnTriggerEnter2D 콜백보다
  원형 판정밖에 못해도 Rigidbody 없이 수백 개를 처리 가능한 B 방식이 유리하다고 판단»

> 결정이 여러 개면 위 블록을 반복. 특정 기능 안의 결정이면 그 기능(양식 ①의 features)에 붙는다고 표시.
