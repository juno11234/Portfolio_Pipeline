# 기술문서 양식 ② — 판단 · 회고 · Bond

> PDF(Bond 포트폴리오)에서 근거가 있는 부분만 뽑아 채운 초안.
> **'왜 골랐나'는 슬라이드에 이미 쓴 본인 판단을 옮긴 것 — 실제 의도와 맞는지 최종 확인·소유할 것.**
> 표기: `**굵게**` · `` `코드` `` · `==채택한 해법==`(파랑) · `!!버그·실패!!`(빨강).
> `« »` = PDF에 측정 근거가 없어 **본인이 직접 채워야 하는 자리**.

---

## 1. 개요 — 왜 (양식 ①의 what 과 짝을 이룬다)

- **why**: 이전 팀 프로젝트가 **높은 결합도와 싱글톤 남용**으로 실행 순서(`Script Execution Order`)에 의존했다. 이를 **인터페이스 + DI(`VContainer`)로 구조 개선**하는 것이 1차 목표. 더해 **AI 코딩 에이전트를 Plan · Build · Review 사이클**로 도입해 프로세스의 실효성을 검증하는 것이 2차 목표.
- **evidence**: « 이전: 싱글톤으로 의존 추적 곤란 · Script Execution Order 강제 -> 인터페이스 참조로 약한 연결 · 의존성 주입 = 연결과정에서 버그나 추적이 크게 필요없음, 인터페이스로 테스트용 Mock 생성해 독자적인 개발가능(이벤트 씬 개발 할때 다른RootScope / MapLifetimeScope 없이 단독으로 이벤트 씬을 실행하기 위한
   Mock 등록 스코프다.) 
- (AI효율성 정적 분석으로 테스트 없이 결함 3건 해결, 21개 모듈 101개 스크립트 규모)»

> 숫자 없는 문제 제기는 근거가 못 된다. evidence는 실측값으로 채운다.

## 2. 고민과 선택

### 결정 ①: 씬 경계로 스테이지 데이터를 어떻게 넘길 것인가  (기능: 맵·원정 플로우 / `StageLoader`)

문제 — `LoadSceneAsync`는 파라미터를 전달하지 못하고, DI 스코프는 씬 경계를 넘지 못한다. 스테이지 씬에 몬스터·이벤트 데이터를 어떻게 넘길 것인가.

- 선택지:
  - A. 싱글톤 · `DontDestroyOnLoad`
  - B. DI 스코프 승계 확장
  - C. 역할 하나짜리 정적 채널 ← **채택**
- 기준별 비교:

  | 기준 | A 싱글톤 | B DI 스코프 승계 | C 정적 채널 |
  |---|---|---|---|
  | 접근 · 구현 | 어디서든 접근, 구현 단순 | 주입 방식의 일관성 유지 | `Set → Read → Clear` 좁은 계약, 쓰는 곳·읽는 곳 명시적 |
  | 상태 수명 · 추적 | 전역 상태라 수명·접근 지점 추적 곤란 | 스코프 수명에 종속 | `Dispose` 시 `Clear` 규약으로 수명 관리 |
  | 씬 경계 · 확장 비용 | 씬은 넘지만 전역 오염 | Additive 조합마다 스코프 설계 복잡, 씬 로드 전달 제약은 그대로 | 정적이라 씬 넘김, 대신 역할 1개 제한 |

- **왜 C를 골랐나**: A의 전역 상태 추적 곤란은 !!이전 팀 프로젝트가 실패한 바로 그 원인!!이라 배제. B는 씬 로드 API의 전달 제약을 그대로 남긴 채 Additive 조합마다 스코프 설계만 복잡해진다. C는 ==역할 1개짜리 정적 채널==로 좁혀 `Set → Read → Clear` 계약을 명시화하고 `Dispose` 시 `Clear`로 수명을 관리 — 정적 상태라는 단점은 역할 1개 제한 + 채널 클래스 격리로 감수했다.

### 결정 ②: 단발 프레임 이펙트를 어떻게 재생할 것인가  (기능: `EffectSpritePlayer`)

문제 — 수십 개의 단발 프레임 이펙트를 스프라이트 세트마다 `Animator` + `Controller` + `.anim` 3종으로 만들 것인가.

- 선택지:
  - A. `Animator` + `AnimationClip`
  - B. Sprite 배열 플레이어 자체 구현 ← **채택**
- 기준별 비교:

  | 기준 | A Animator + Clip | B Sprite 배열 플레이어 |
  |---|---|---|
  | 자산 관리 | 이펙트마다 자산 3종 증식, 관리·에셋 부담 | 인스펙터에 `Sprite[]` 배열만, 자산 부담 제거 |
  | 일시정지 중 재생 | `timeScale = 0` 재생에 별도 설정 필요 | unscaled 재생, 일시정지 연출 중에도 이펙트만 재생 |
  | 아티스트 확인 | 표준 워크플로(블렌딩·전이·이벤트 지원) | 에디터 프리뷰로 플레이 없이 확인 |
  | 한계 | 단발 재생에 스테이트 머신은 과한 구조 | 블렌딩·전이 없음, 단순 프레임 이펙트로 용도 한정 |

- **왜 B를 골랐나**: 대상이 ==단발 프레임 이펙트==라 스테이트 머신은 과한 구조. 이펙트마다 자산 3종이 증식하는 부담과 `timeScale = 0`(일시정지 연출) 대응의 번거로움을 피하려 `Sprite[]` 배열만 받는 플레이어를 직접 구현. 블렌딩·전이가 없다는 한계는 용도를 단순 프레임 이펙트로 한정해 감수했다.

### 결정 ③: 완료 콜백 채널 — AI 제안을 뒤집은 역제안  (기능: `StageCompletionChannel`)

문제 — `StageCompletionChannel`이 단일 슬롯이라, 전투 중 컷씬이 뜨면 !!컷씬 콜백이 전투 콜백을 덮어써 컷씬 후 맵으로 복귀하지 못했다!!.

- 선택지:
  - A. (AI 제안) 컷씬 언로드 후 전투 콜백 재등록
  - B. (AI 제안) 컷씬 전용 정적 채널 신설
  - C. (내 결정) 단일 슬롯 → `Stack` 채널 ← **채택**
- 기준별 비교:

  | 기준 | A 콜백 재등록 | B 전용 채널 신설 | C Stack 채널 |
  |---|---|---|---|
  | 씬 중첩 대응 | 되돌리기 로직이 호출부마다 산재, 중첩 깊어지면 복원 순서 관리 복잡 | 씬 구조 늘 때마다 채널 클래스 증식 | 최근 씬부터 `Pop`, 씬이 더 중첩돼도 그대로 동작 |
  | 기존 코드 영향 | 호출부마다 복원 로직 추가 | 같은 코드 중복 | 기존 호출부 수정 최소, `Register`/`Invoke` 시그니처 유지 |

- **왜 C를 골랐나**: AI가 제안한 A·B를 그대로 받지 않고 뒤집었다. A는 되돌리기 로직이 호출부마다 산재해 중첩이 깊어질수록 복원 순서 관리가 복잡해지고, B는 씬 구조가 늘 때마다 채널 클래스가 증식해 같은 코드가 중복된다. 대신 단일 슬롯을 ==`Stack<Action<StageResult>>`==로 바꿔 최근 씬부터 `Pop`하게 하면 씬이 더 중첩돼도 흐름이 그대로 맞아떨어지고, `Register`/`Invoke` 시그니처를 유지해 기존 호출부 수정이 최소가 된다. (`Invoke`는 `Peek().Invoke()`로 제거 없이 실행, 언로드 시 `Pop`)
