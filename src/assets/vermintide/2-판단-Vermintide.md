# 기술문서 양식 ② — 판단 · 회고 (Vermintide)

> **채운 근거:** 양식 ①(구현) + 버민타이드 PDF(신·구버전)에 **본인이 남긴 판단·회고**를 이 양식 구조로 정리한 것.
> AI 가 새로 지어낸 결정이 아니라, 두 문서에 이미 있는 근거를 옮겨 담았다. **실제 '왜'는 본인이 대조·보강**할 것(→ 맨 아래 부록).
> **강조 규칙:** `**굵게**` · `` `코드` `` · `==해법==`(파랑 = 내가 고른 해법 · 기제) · `!!실패!!`(빨강 = 버그 · 실패).
> 파일 · 스크린샷 · 링크는 여기 말고 **양식 ③**.

---

## 1. 개요 — 왜 (양식 ①의 what 과 짝)

- **why**: Vermintide 모티브라 화면을 가득 채우는 **대규모 고블린 웨이브** 위에 무기 3종 · 고블린 4종 · 보스 1종이 얹혀야 했다. 그래서 이 프로젝트의 설계 목표는 두 가지였다 — (1) 무기 · 몬스터 · 보스 상태를 **기존 코드 수정 없이 종류만 늘릴 수 있는 확장 구조**로 짜는 것, (2) 매 프레임 쏟아지는 피격 처리가 서로 얽히지 않고 **프레임 스파이크 없이** 흐르게 하는 것. 그래서 객체끼리 직접 호출하는 대신 인터페이스(`IFighter`/`IInteractable`/`IObjectPoolItem`) · 추상 클래스(무기 · 몬스터 · 보스 상태) · 이벤트 큐로 결합을 끊는 방향을 택했다.

- **evidence**: (실측된 문제 — 숫자 있는 것만)
  - 런타임 프레임이 **15FPS 이하**로 떨어지는 구간 발생. 프로파일링 결과 `Gfx.WaitForPresent` 가 잡혀 **CPU 가 GPU 작업을 대기**하는 GPU 병목이 원인 (Unity Profiler).
  - 카메라를 급격히 돌리면 한 스윙에 콜라이더가 대상에 **두 번 닿아 데미지가 중복 적용**되는 버그.
  - 전투 이벤트는 프레임당 처리량을 상한(`MAX_EVENT_PROCESS_COUNT` = 10)으로 제한해 폭주 프레임을 나눠 처리하도록 설계.

## 2. 고민과 선택 (핵심 기능과 분리된 독립 섹션)

> 결정마다 선택지(2개+, 하나만 채택) · 같은 잣대로 비교 · 왜 골랐나 를 반복. 아래는 구현 문서와 슬라이드에 근거가 남아 있는 5건.

### 결정 1: 전투 처리 — 직접 호출 vs 이벤트 큐  *(양식 ① 기능 1)*
- 선택지:
  - A. 공격 주체가 피격 대상의 `TakeDamage()` 를 **직접 호출**
  - B. `CombatEvents` 를 만들어 `CombatSystem` 의 `Queue` 에 넣고 `CombatSystem.Update` 가 꺼내 처리 ← **채택**
- 기준별 비교:

  | 기준 | A. 직접 호출 | B. 이벤트 큐 |
  |---|---|---|
  | 결합도 | 공격자가 피격 대상을 직접 참조 → 서로를 알아야 함 | 공격자 · 피격자 모두 `CombatSystem` 만 알면 됨, 서로 모름 |
  | 처리 타이밍 | 접촉 즉시(지연 0) | 다음 `Update` 프레임(1프레임 지연 가능) |
  | 프레임 부하 제어 | 한 프레임에 타격이 몰리면 그대로 다 처리 | `MAX_EVENT_PROCESS_COUNT`(10)로 프레임당 처리량 상한 |
  | 부가 반응(이펙트·UI) 확장 | 공격 코드가 혈흔·게이지·카운트를 직접 호출 → 로직에 침투 | `Events` 브로드캐스트를 **구독**만 하면 역참조 없이 반응 |

- **왜 B 를 골랐나**: 피격 한 번에 혈흔 · 스킬 게이지 · 웨이브 카운트 같은 반응이 여럿 붙는데, 이걸 공격 코드가 다 직접 호출하면 전투 로직이 표현 · UI 코드에 얽힌다. ==큐 + 이벤트 브로드캐스트== 로 바꾸니 구독자들이 서로를 몰라도 각자 반응했고, 처리량 상한으로 폭주 프레임도 나눠 처리할 수 있었다. **즉시성(1프레임 지연)을 내주고 결합도와 프레임 안정성을 택한** 것.

### 결정 2: 무기 구조 — Player 직접 참조 vs 전략 패턴(WeaponBase)  *(양식 ① 기능 2 · PDF "상속의 개념")*
- 선택지:
  - A. `Player` 가 `GreatSword`·`Gun`·`Potion` 을 각각 필드로 **직접 참조**
  - B. `WeaponBase` 추상 타입 하나(`currentWeapon`)로 다루고 무기별 행동은 override ← **채택**
- 기준별 비교:

  | 기준 | A. 직접 참조 | B. 전략 패턴(WeaponBase) |
  |---|---|---|
  | 무기 추가 비용 | 새 무기마다 `Player` 에 필드·분기 추가 | `WeaponBase` 파생 클래스만 추가, `Player` 무수정 |
  | 다형성 | 무기별로 다른 호출부 필요 | `currentWeapon.RMBClick()` 하나로 통일 |
  | 무기별 특수행동(가드 등) | 어차피 `Player` 가 전부 앎 | 대부분 부모 타입으로 처리, 가드만 `is GreatSword` 로 최소 노출 |
  | 애니메이터 교체 | 무기별 코드 분기 | 무기가 든 `RuntimeAnimatorController` 로 일괄 교체 |

- **왜 B 를 골랐나**: 무기를 늘릴 때마다 `Player` 를 뜯어고치는 게 싫었다. `WeaponBase` 로 추상화하니 ==부모 타입에만 의존== 하게 되어 새 무기는 클래스 추가로 끝났다. 

### 결정 3: 렌더 최적화 — Occlusion Culling 적용 여부  *(PDF "GPU 병목 진단")*
- 선택지:
  - A. **미적용** — 컬링 판정용 CPU 연산 비용을 아낌
  - B. Static 배칭 + Occlusion Culling **적용** ← **채택**
- 기준별 비교:

  | 기준 | A. 미적용 | B. 적용 |
  |---|---|---|
  | CPU 비용 | 컬링 판정 연산 없음 | 동적 오브젝트 가림 판정 연산이 추가로 듦 |
  | GPU 렌더 부하 | 안 보이는 메쉬도 렌더 | 비가시 메쉬 렌더 차단으로 GPU 부하 ↓ |
  | 씬 특성 적합도 | 개활지처럼 가림이 적으면 이득 작음 | **겹치는 오브젝트가 많은 실내 던전** 에서 이득 큼 |
  | 결과 | 15FPS 병목 유지 | 15FPS → 60+FPS |

- **왜 B 를 골랐나**: 프로파일러에서 `Gfx.WaitForPresent` 로 !!GPU 병목!! 을 확인했다. Occlusion Culling 은 공짜가 아니라 **가림 판정용 CPU 연산이 추가로 드는** 트레이드오프였지만, 이 게임은 겹치는 오브젝트가 많은 실내 던전이라 컬링으로 얻는 GPU 이득이 CPU 비용보다 크다고 판단해 적용했다. 동시에 Static 오브젝트는 드로우콜을 묶고, Occlusion Culling 이 참조하는 베이크 데이터도 Static 기준으로 생성되므로 함께 켰다. 결과 ==15FPS → 60+FPS==.

### 결정 4: 점프력 계산 — 직접 velocity 지정 vs Sqrt(미터 단위 환산)  *(구버전 PDF "기본 조작")*
- 선택지:
  - A. `velocity.y` 에 raw 속도값을 **직접 지정** — Sqrt 연산 없음
  - B. `velocity.y = Mathf.Sqrt(jumpHeight * -gravity …)` 로 **미터 단위 `jumpHeight` 를 속도로 환산** ← **채택**
- 기준별 비교:

  | 기준 | A. 직접 velocity 지정 | B. Sqrt 환산 |
  |---|---|---|
  | 연산 비용 | Sqrt 없음 (가벼움) | 점프 입력마다 `Sqrt` 1회 |
  | 변수 가독성 | raw 속도값을 감으로 튜닝 | 미터 단위 `jumpHeight` 로 "몇 m 점프" 직관적 설정 |
  | 성능 영향 | 없음 | **입력당 1회뿐** 이라 사실상 무시 가능 |

- **왜 B 를 골랐나**: `Sqrt` 가 무거운 연산인 건 맞다. 하지만 점프는 매 프레임이 아니라 **점프 입력이 있을 때 한 번만** 호출되므로 성능 영향이 사실상 없다고 판단했다. 대신 점프력을 ==미터 단위(`jumpHeight`)== 로 다룰 수 있어 "2m 높이로 점프" 같은 직관적 튜닝이 가능해지는 가독성을 택했다. **연산 비용을 호출 빈도로 나눠서** 본 판단.

### 결정 5: 중복 타격 필터 — 대상별 bool 플래그 vs 피격 집합(HashSet)  *(PDF "HashSet 중복 타격 필터링")*
- 배경: 콜라이더 활성 중 카메라를 급회전하면 한 스윙에 콜라이더가 대상에 !!두 번 닿아 데미지가 중복 적용!! 됐다. 한 스윙에 같은 대상은 한 번만 맞아야 한다.
- 선택지:
  - A. **대상별 bool 플래그** — 대상마다 "이미 맞음" bool 을 두고, 맞으면 `true` 로 바꿔 재피격 무시, 나중에 되돌림
  - B. **피격 집합(HashSet)** — 공격자가 이번 스윙에 맞은 대상을 `HashSet` 에 담아 재피격 무시, 스윙(콜라이더 활성)마다 `Clear` ← **채택**
- 기준별 비교:

  | 기준 | A. 대상별 bool 플래그 | B. 피격 집합(HashSet) |
  |---|---|---|
  | 상태 위치 | 대상마다 "이미 맞음" bool 을 각자 보유 | 공격자 한 곳의 `HashSet` 에 집중 |
  | 재피격 판정 | 대상의 bool 확인 (필드 접근, A 가 더 쌈) | `Contains` O(1) |
  | 초기화(스윙 종료) | 맞은 대상들의 bool 을 되돌려야 함 → 어떤 대상을 맞혔는지 따로 기억 필요(사실상 컬렉션) | 공격자에서 `Clear` 한 번 |
  | 책임 소재 | "내 스윙에 맞았나" 상태가 대상 쪽에 흩어짐 | "이번 스윙에 내가 맞힌 대상"을 공격자가 소유 |

- **왜 B 를 골랐나**: 판정 자체는 bool 필드 접근이 더 싸다. 문제는 **초기화**였다 — 대상별 bool 은 스윙이 끝날 때 맞은 대상들의 플래그를 되돌려야 하는데, 그러려면 어떤 대상을 맞혔는지 따로 기억해야 해서 결국 컬렉션이 필요해진다(아니면 전체 대상을 순회해 초기화). ==HashSet== 은 "이번 스윙에 내가 맞힌 대상"을 공격자가 한 곳에 들고 있다가 스윙 시작 `OnEnable` 에서 `Clear` 한 번으로 리셋되므로 상태의 소유와 초기화가 깔끔하다. **판정 비용을 조금 내주고 상태 관리·초기화의 단순함을 택한 것.**

## 부록 — 사람이 확인 · 보강할 곳 (본인)

- **evidence 수치**: `15FPS → 60+FPS` 외의 정량 전후 값(GC 할당 · 드로우콜 등)은 소스에 없다(양식 ① 부록도 동일 지적). 실측 자료가 있으면 1번 evidence 에 채우고, **없으면 이 한 건으로만 유지 — 숫자는 지어내지 말 것.**
- **결정의 '왜'**: 위 5개 결정의 비교표 · 선택 이유는 구현 문서와 슬라이드에 남긴 근거로 정리한 것이다. 면접에서 그대로 말하기 전에 **실제로 그때 무엇을 보고 골랐는지 본인 기억과 대조**해 다듬을 것. 특히 결정 1의 "프레임당 10건 상한"이 실전에서 실제로 발동될 만큼 이벤트가 몰린 구간이 있었는지 — 있으면 그 장면을, 없으면 '설계상 안전장치'로 표현.
- **결정 5(HashSet)의 선택지 A**: '대상별 bool 플래그'는 가장 흔한 대안으로 정리한 것. 실제로 이 방식과 저울질했는지 본인 확인 — 안 했다면 A 를 "BEFORE: 중복 방지 없이 트리거마다 데미지"로 바꿔도 됨(PDF '문제' 서술과 일치). 비교표의 'A 는 판정이 더 싸지만 초기화가 문제'라는 논지는 실제 코드(`Clear` 가 `OnEnable` 에 있음)와 맞음.
- **결정 4(점프 Sqrt)의 실제 코드**: 구버전 슬라이드 기준으로 공식을 `Sqrt(jumpHeight * -gravity)` 로 적었는데, 실제 코드의 상수(2 곱 유무 등)와 비교해 표현을 맞출 것.
- **improvements 축소**: 다음 계획을 `RemoveMonster` 자동화 한 건으로 줄였다. 이전 후보(싱글턴→DI, 최적화 지표 남기기, 상태·무기 데이터 외부화)는 필요하면 되살릴 수 있음 — 특히 싱글턴→DI 는 이후 방향과 맞아 강한 카드였음.
