개요

1인 개발 60 스크립트 · 5,484줄 Unity 6000.0.56f1 · URP C# · Rider 3인칭 소울라이크 액션

스태미나를 소모하는 회피·공격과 화톳불 세이브, 2페이즈 보스를 갖춘 3인칭 소울라이크 액션 게임입니다. 1인 개발로 FSM 플레이어 · Behaviour Tree 몬스터 AI · 이벤트 기반 MVVM · Jenkins CI까지 60개 스크립트(5,484줄)를 직접 설계·구현했습니다.

WHAT
플레이어는 유한 상태 머신(FSM)으로, 몬스터는 행동 트리(BT)로 움직입니다. 전투·피격·UI 갱신은 모두 이벤트와 Observable로 흐르는 단방향 구조입니다.
WHY
상태가 늘수록 전이 조건이 폭발하는 FSM의 한계를 항상 트리 구조인 BT로 풀고, 잦은 UI 변경에는 낮은 결합의 MVVM으로 대응하는 것이 이 프로젝트의 승부처입니다.
▶ 10초 요약 — 이 문서가 증명하는 것
입력→상태→전투→데이터→UI 단방향 계층을 두고, 역방향은 전부 event · Observable<T>로 통신합니다. 몬스터는 재귀 Behaviour Tree로, 그중 Sequence_Memory가 콤보 끊김 버그를 해결했습니다. 전투는 이벤트 큐 + 애니메이션 주도 판정, 빌드는 Jenkins CI로 자동화했습니다.

전투 · MVVM

모든 피해·회복은 CombatSystem 싱글턴의 이벤트 큐를 거쳐 결정론적으로 처리되고, 체력·스태미나 UI는 Observable<T>를 구독하는 MVVM으로만 갱신됩니다.

핵심 동작 이벤트 큐 — 프레임당 10건

무기의 OnTriggerEnter는 피해를 즉시 적용하지 않고 큐에 쌓습니다. Update가 프레임당 최대 10건만 처리해 순서를 결정론적으로 만들고 부하를 분산합니다.

private const int Max_Event_Count = 10;

void Update()
{
    int processCount = 0;
    while (_eventQueue.Count > 0 && processCount < Max_Event_Count)
    {
        InGameEvent e = _eventQueue.Dequeue();
        switch (e.Type)
        {
            case EventType.Combat: e.Receiver.TakeDamage((CombatEvent)e); break;
            case EventType.Heal:   e.Receiver.TakeHeal((HealthEvent)e);  break;
        }
        processCount++;
    }
    // 남은 이벤트는 다음 프레임으로 이월 — 한 프레임 부하 상한
}

Observable.cs · 값 변경 통지 + 구독 즉시 동기화
public T Value
{
    get => _value;
    set
    {
        // 같은 값이면 통지하지 않음 — 중복 갱신 차단
        if (EqualityComparer<T>.Default.Equals(_value, value)) return;
        _value = value;
        OnValueChanged?.Invoke(_value);
    }
}

public void Subscribe(Action<T> listener)
{
    OnValueChanged += listener;
    listener?.Invoke(_value);   // 구독 즉시 현재값 1회 전달 → 초기화 코드 불필요
}

평가 규칙 Selector(OR) vs Sequence(AND)

Selector — 우선순위 (OR)
자식을 순서대로 평가하다 하나라도 Success/Running이면 즉시 그 결과 반환. 전부 Failure여야 Failure. → 첫 성공 분기를 고르는 우선순위 스위치
Sequence — 연속 행동 (AND)
자식을 순서대로 평가하다 하나라도 Failure면 즉시 중단. 전부 통과해야 Success. → 조건·행동을 한 줄로 잇는 시퀀스

핵심 이슈 콤보가 끊기는 버그 — Sequence → Sequence_Memory

문제매 프레임 처음부터 재평가
적이 콤보 도중 플레이어가 사거리를 벗어나면, Sequence가 매 프레임 앞 조건(CheckAttackRange)부터 재평가해 실패 → 진행 중이던 콤보가 즉시 끊깁니다.
해결실행 위치를 기억
_lastRunningChildIndex로 Running이던 자식 인덱스를 기억했다가, 다음 프레임에 그 자리에서 재개합니다. 진입 조건을 다시 통과할 필요 없이 진행 중인 타부터 이어집니다.
결과 3타 콤보·후퇴 시퀀스 등 중단되면 안 되는 연속 행동은 전부 Sequence_Memory로 조립했습니다. 순수 조건 검사만 필요한 곳은 기존 Sequence를 유지합니다.
Selector.cs · Sequence_Memory.Evaluate()
public override NodeState Evaluate()
{
    // 이전 평가가 Running이 아니었다면 처음부터
    if (State != NodeState.Running) _lastRunningChildIndex = 0;

    for (int i = _lastRunningChildIndex; i < Children.Count; i++)
    {
        switch (Children[i].Evaluate())
        {
            case NodeState.Failure:
                _lastRunningChildIndex = 0;
                return State = NodeState.Failure;
            case NodeState.Running:
                _lastRunningChildIndex = i;   // 실행 위치 기억 → 다음 프레임 재개
                return State = NodeState.Running;
            case NodeState.Success:
                continue;                     // 다음 자식으로
        }
    }
    _lastRunningChildIndex = 0;
    return State = NodeState.Success;
}

시점 skillCool은 애니메이션 완료가 아니라 조건 검사 통과 즉시 true가 됩니다. 즉 15초 쿨다운에 시전 시간이 포함되어, 실제 재사용 간격은 15초 − (선딜+타격+후딜)입니다. OnSecondPhase는 절반 이하 모든 피격마다 반복 발행되지만 수신 측이 멱등해 문제되지 않습니다.

같은 원리의 노드 중단
Sequence와 Selector 모두 노드가 중단될 때 정리할 방법이 없어 같은 원인의 문제를 겪었지만, 각각 Sequence_Memory와 Cleaner라는 서로 다른 방법으로 해결했습니다.
시각화의 가치
코드로는 추적이 힘들던 행동 트리의 문제 지점이, 기술 문서를 위해 그린 다이어그램에서는 한눈에 보였습니다. 도식화하며 디버깅했다면 시간을 크게 줄일 수 있었을 것입니다.
데이터 주도 BT의 가능성
노드를 enum으로 분류하는 과정에서, 트리 구조 자체를 데이터 테이블화해 코드 수정 없이 AI 패턴을 조립하는 구조로 확장할 수 있겠다고 느꼈습니다.
