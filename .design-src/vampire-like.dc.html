개요

1인 개발 2026.03.03–03.18 · 2주 Unity 6000.0.56f1 C# · Rider
WHAT
수백 마리의 적을 자동 공격으로 처치하고, 경험치를 모아 레벨업하며 스킬 3종 중 하나를 선택·강화하는 2D 탑다운 뱀서라이크. 플레이어는 이동만 조작합니다.
WHY
대량 오브젝트를 안정적으로 처리하기 위한 GC 최적화가 핵심 목표. 프레임을 멈추게 하는 GC 스파이크를 구조적으로 제거하는 것이 이 프로젝트의 승부처입니다.
▶ 10초 요약 — 이 문서가 증명하는 것
이벤트 큐 + 오브젝트 풀 + 무할당 물리 쿼리로 프레임당 GC 할당을 196회 → 4회(97%↓), Material 접근을 824회 → 249회(70%↓)로 줄여, 화면을 가득 채운 적 앞에서도 스파이크 없는 프레임을 확보했습니다.

흐름 초기화 시퀀스 — 의존 순서를 await로 직렬화

모든 데미지는 InGameEvent 구조체로 큐에 쌓이고, CombatSystem.Tick()이 한 프레임에 몰아서 처리합니다.

CombatSystem.cs · Tick()
public void Tick()
{
    while (_eventQueue.Count > 0)   // 큐가 빌 때까지 한 프레임에 모두 소비
    {
        var e = _eventQueue.Dequeue();
        switch (e.Type)
        {
            case EventType.Combat: e.Receiver.TakeDamage(e); break;
            case EventType.Heal:   e.Receiver.Heal(e);       break;
        }
    }
}

SkillBase.cs · LevelUp()
public virtual void LevelUp(SkillData skillData)
{
    if (CurrentLevel >= SkillData.maxLevel) return;   // 최대 레벨 가드
    Damage  += SkillData.atkPerLevel;                 // 데미지 상승
    Cooldown = Mathf.Max(0.1f, Cooldown - 0.5f);      // 쿨타임 감소 (하한 0.1s)
    CurrentLevel++;
}

결정 1 전투 처리 — 직접 호출 vs 이벤트 큐

A 직접 호출
B 중앙 처리 이벤트 큐채택
결합도·확장성
공격자가 대상을 직접 호출, 스킬마다 처리지점 분산
AddInGameEvent만 알면 됨, 신규 스킬 추가해도 무변경
처리 타이밍
같은 프레임 즉시 반영
최대 1프레임 지연
반환값 활용
데미지·오버킬·사망여부 즉시 리턴
리턴값 없음 → 역이벤트/콜백 필요
순회·재진입 안전성
타격 중 사망 → 컬렉션 변경 유발
때리는 시점 ≠ 죽는 시점으로 분리
결정 → B 직접 호출은 ‘타격 도중 적 사망 → 풀 반납·컬렉션 변경’이 중첩돼 오류 위험이 큽니다. 이벤트 큐는 이를 구조적으로 차단하고 느슨한 결합과 확장을 얻습니다. 타이밍 지연은 미미하고 반환값은 콜백으로 보완 가능하다 판단해 B를 선택했습니다.

결정 2 충돌 판정 — OnTriggerEnter2D vs Raycast & OverlapCircle

그 밖의 결정 힙 할당을 없애기 위한 선택들

무할당 물리 쿼리
ContactFilter2D + 사전 할당 배열을 재활용해 매 프레임 신규 배열 생성을 없앰. 거리 비교는 sqrMagnitude로 제곱근 회피.
풀 반납 콜백 캐싱
몬스터 사망 시 풀 반납 람다를 딕셔너리에 캐싱해, 매 죽음마다 클로저를 새로 할당하는 것을 방지.
MaterialPropertyBlock
피격 플래시에서 머티리얼 인스턴스를 만들지 않고 프로퍼티(_Amount)만 덮어써 공유 머티리얼 유지.
struct InGameEvent
전투 이벤트를 클래스가 아닌 구조체로 설계해, 초당 수백 건이 발생해도 힙 할당이 생기지 않도록 함.

배운 점 Troubleshooting

① 평균이 아니라 스파이크를 제거하는 것
의문점
그래픽 병목을 잡았을 때와 달리 평균 프레임이 유의미하게 늘지 않았다.
원인 · Boehm GC
힙에 가비지를 쌓다 임계치에서 수거하며, 이때 게임 로직이 멈춘다(스파이크).
알게된 점
목적은 평균 FPS가 아니라 GC 호출을 줄여 스파이크를 없애는 것. 가비지가 많을수록 더 자주·오래 멈춘다.
② 텍스처 압축도 트레이드오프의 대상
문제
에디터에서 선명하던 픽셀아트가 Android 빌드 후 색이 뭉개짐.
해결
RGBA 32bit 무압축 적용. 픽셀아트는 해상도가 낮아 메모리 영향이 미미.
배운 점
안드로이드라고 무조건 압축이 아니라, 그래픽 스타일에 따라 압축도 선택의 문제.
BEFOREASTC 6×6 압축
색이 뭉개지고 픽셀 경계가 흐려짐
AFTERRGBA 32bit 무압축
선명한 픽셀 경계 복원 · 저해상도라 메모리 영향 미미

기술 부채 정의되어 있으나 런타임 미사용
